## Successes and Strengths

1. **Sprint change proposal absorbed cleanly** — Removing TanStack Query + ky in favor of native fetch + custom ETag caching was a significant mid-planning pivot. The formal proposal process with impact analysis worked well. The result is a leaner, more controlled architecture with zero external runtime dependencies beyond `@microsoft/signalr` and `ulidx`.

2. **Ports-and-adapters architecture (Story 2.6)** — MockCommandBus, MockQueryBus, and MockSignalRHub implement the same interfaces as real implementations, with contract tests validating behavioral parity. This is architectural insurance — any future implementation swap is automatically validated.

3. **Graceful SignalR degradation (Story 2.7)** — SignalR is purely additive. If the connection fails, queries continue via polling. ConnectionStateProvider tracks HTTP reachability independently. The `start()` failure is silent. Module developers never think about transport selection.

4. **Massive meaningful test growth** — 80 → 326 tests. Not surface-level tests — includes ETag 304 no-flicker assertions, retry backoff with fake timers, 50-group SignalR limit boundary tests, and fail-closed authorization.

5. **Quality gate story pattern (Story 2.8)** — Audit-first approach validated the complete ETag integration chain across all invalidation sources: manual refetch, command-complete, SignalR push, polling, and window focus. No new production code — pure verification. High value for integration-heavy sequences.

6. **Fix-forward policy worked** — Zero cascading test failures across 9 stories. Each story maintained all prior tests passing. This was an Epic 1 action item that delivered.

7. **CommandEventBus for projection invalidation** — Domain-level invalidation triggers (command-complete events) propagate through the event bus to invalidate relevant query caches. Clean separation between command and query concerns.

8. **Dual correlation ID documentation (Story 2.1)** — Upfront documentation of the request ULID vs response GUID system prevented confusion in all subsequent stories.

## Challenges and Growth Areas

1. **0% first-pass review rate persists** — All 9 stories required post-review fixes. Common findings: test methodology issues (real vs fake timers in 2.5), edge case handling (429 special-casing in 2.2, colon-in-key delimiter in 2.7), semantic correctness (`isLoading` on cache hits in 2.9). A pre-review checklist was committed in Epic 1 but not formalized.

2. **Story 2.2 wasted review cycle** — Review was triggered before implementation existed, returning "Blocked — implementation missing." This is a sequencing gap in the review process.

3. **`activeTenant` type assumption wrong in story specs** — Story 2.3 discovered that `activeTenant` is `string | null`, not `{ id: string }` as specs assumed. All hooks had to be adapted at implementation time. Story specs should verify type assumptions against actual code during story creation.

4. **`packages/` in `.gitignore`** — Story 2.7 discovered that git cannot track or audit files under `packages/cqrs-client/`. Story file lists are unverifiable via git. Determine if intentional or accidental.

5. **Git hooks still missing (carried from Epic 1)** — Import ordering violations hit Stories 2.5, 2.6. Pre-existing import-order errors remain in untouched test files. Two retrospectives have committed to this; two retrospectives have failed to deliver.

6. **Tech debt backlog never consolidated (carried from Epic 1)** — Debt items remain scattered across 9 story files. No single source of truth for prioritization.

## Key Insights

1. **Quality gate stories are high-value** — Story 2.8 (zero production code, pure audit + test) caught integration issues that per-story testing missed. Use after integration-heavy sequences.

2. **Story intelligence docs remain the primary knowledge transfer mechanism** — A single reference doc would amplify this further.

3. **Sprint change proposals need a formal process** — The mid-planning pivot worked because it had clear impact analysis. Formalize this.

4. **Type assumptions in story specs must be verified against code** — Spec-time discovery is cheaper than implementation-time discovery.

5. **Carried action items erode retrospective credibility** — Two items on their second carry signal a systemic issue with follow-through.
